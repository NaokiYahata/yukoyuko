# project-init — プロジェクト初期化

GitHub Template から作られた直後のリポジトリを、具体的なプロジェクトとして立ち上げる。
**開始前に `wiki/_schema.md` に目を通すこと。**

## 手順

1. **日付プレースホルダを置換する（最初に実行）**
   - テンプレートのシードには `{{DATE}}` が埋め込まれている。これを当日の実日付（YYYY-MM-DD）へ一括置換する:
     ```
     grep -rl '{{DATE}}' wiki | xargs perl -i -pe 's/\{\{DATE\}\}/<今日の日付>/g'
     ```
   - `wiki/_templates/` 内の `{{DATE}}` は**置換しない**（雛形は今後も使い回すため、ページ生成時に都度置換する）。上記コマンドは `_templates/` を除外するか、対象を明示すること。

2. **ヒアリング** — 次を対話で聞き出す（分かる範囲でよい、後から更新できると伝える）:
   - プロジェクト名 / 一行説明
   - 背景・解きたい課題（なぜやるのか）
   - ゴールと成功の定義（測れる指標があれば）
   - 主要ステークホルダー / 関係者
   - 想定期間・マイルストーン
   - 既にあるソースの種類（議事録・仕様書・Slack・調査資料など）

3. **最初のリリースと backlog を立ち上げる** — このテンプレートはイテレーティブ（リリースモデル）。最初に全部を決めない。
   - 直近のリリース予定（月 `YYYY-MM`）をヒアリングする。決まっていれば `wiki/releases/YYYY-MM.md` を雛形（`wiki/_templates/release.md`）から作成し、分かっている範囲でゴールとスコープを記入。
   - 把握済みの要求・やりたいことは `wiki/backlog.md` に1行ずつ初期投入する（詳細化は不要、粗くてよい）。

4. **CLAUDE.md「プロジェクト固有設定」を更新** — ヒアリング結果で各項目を埋める。**現在のリリースサイクルはここに書かない**（overview.md が唯一の記録場所）。

5. **overview.md を更新** — `wiki/overview.md` の一行サマリ・背景・ゴール・主要論点を埋める。`status` を `active` に。「現在のリリースサイクル」に直近リリース（`[[YYYY-MM]]`、未定なら未設定のまま）を記入。判明していれば主要ステークホルダーを `entities/`（雛形 `wiki/_templates/entity.md`）にページ化し index に追加。

6. **README を任意で調整** — テンプレート説明をプロジェクト固有の説明に置き換えてよいか確認する。

7. **検索（qmd）のセットアップを案内** — Wikiが育つと検索が要るため、`/qmd-setup` の実行を勧める（任意・推奨）。導入したら CLAUDE.md「プロジェクト固有設定」に「検索: qmd 有効」と記録。

8. **log に追記**:
   ```
   ## [YYYY-MM-DD] note | プロジェクト初期化: <プロジェクト名>
   - ゴール: ...
   - 次のアクション: 資料を raw/_originals/ に置いて /wiki-ingest
   ```

9. **次の一歩を案内** — 「資料（pptx/pdf/xlsx/docx 等）を `raw/_originals/` に置いて `/wiki-ingest` で取り込みましょう」と促す。

## 原則
- 不明点は空欄のままにせず「未確認」と明示し、後で埋める前提にする。
- 日付は今日の実際の絶対日付を使う。
