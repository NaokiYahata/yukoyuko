# wiki-sync — 手編集・実装変更の取り込み

人間がエディタで `wiki/` を直接書き換えた、または **このリポジトリの実装コードを変更した** あと、その変更を **Wiki全体に織り込み直す**。`/wiki-ingest` が raw 起点なのに対し、これは **既存の手編集・コード変更起点**。変更そのものだけでなく、リンク・index・関連ページ・整合性まで波及させるのが目的。

**開始前に必ず `wiki/_schema.md`（規約）と `wiki/index.md`（現在の構造）を読むこと。**

## このリポジトリと実装コード

このテンプレートは、Wiki運用に加えて **アプリケーション実装そのもののリポジトリ** にもなりうる。その際の役割分担:

- **Wiki（`wiki/`）= 知識**: なぜ作るか／何を決めたか（decisions/ADR）・設計概念（concepts）・リリーススコープ（releases/overview/backlog）。
- **実装コード = 「実際に作られたもの」の真実**: コードが動作の最終的な出所。
- wiki-sync の仕事は両者の **ドリフト（食い違い・陳腐化）の検出と橋渡し**。コードの正しさ自体のレビューは `/code-review` の仕事で、ここでは扱わない。
- **コード扱いの境界**: `wiki/` `raw/` `docs/` `.claude/` とルートのメタファイル（`README.md` `CLAUDE.md` `.gitignore` 等）以外の追跡ファイルを「実装コード」とみなす。コードのビルド/テスト/規約は `CLAUDE.md` の「コードベースの規約」セクションに従う。

## 手順

### 1. 変更を検出して仕分ける
- リポジトリ全体の作業ツリー差分を見る:
  ```bash
  git status --porcelain               # M=変更 A=追加 D=削除 R=改名 ??=未追跡
  git diff                             # 変更本文の中身
  ```
- 変更を3つに仕分ける: **(a) Wiki編集**（`wiki/` 配下）/ **(b) 実装コード変更**（上記の境界に従う）/ **(c) その他**（docs・設定など、必要に応じて扱う）。
- 作業ツリーに差分が無く「もうコミット済み」と言われた場合は、基準コミットを確認して `git diff <ref>`（例 `HEAD~1`、または前回 sync 以降）で差分を取る。

### 2. 変更内容を確認する（人間と協調）
- 変更（Wiki編集・コード変更それぞれ）と diff を読み、「何が・なぜ変わったか」を3〜6点で要約して提示し、解釈と波及方針を確認する。一括反映を明示的に指示されていればこの確認は省略してよい。

### 3. Wiki編集を波及させる（(a) があるとき）
- **双方向リンク**: 新規ページは関連ページからの被リンクを張り `index.md` に追加。改名は `[[旧名]]→[[新名]]` を全置換し index 更新（`grep -rn '\[\[旧名\]\]' wiki`）。削除はダングリングリンクの差し替え/撤去と index からの除去。本文に増えた `[[...]]` はリンク先の実在を確認し、可能なら相互リンク。パス付き等の規約違反リンクは正規化。
- **矛盾**: 編集後の主張を関連ページと突き合わせ、**食い違いは消さずに明示** し、どちらがより新しい/信頼できるか根拠を添える。
- リリース・backlog・overview への影響を反映（スコープ/状態変化 → `releases/YYYY-MM.md`・`release:` タグ・`overview.md`、新規要求 → `backlog.md`）。

### 4. 実装コードとWikiの整合をとる（(b) があるとき）
- **コード変更が触れた領域に対応するWikiページを特定** する。手がかり: ファイルパス/モジュール名/機能名で `grep -rn '<名前>' wiki`、関連する `entities/`（システム・機能）・`concepts/`（設計）・`decisions/`（ADR）・現在の `releases/YYYY-MM.md` を当たる。
- **ドリフトを検出する**:
  - コードが ADR/concept の記述と食い違う → どちらが現実かを切り分ける。コードが新しい現実なら該当ページを更新（必要なら新しい意思決定として `decisions/` にADRドラフト、`<op>` = `decision`）。意図しない逸脱の疑いなら**消さずに矛盾として明示** し人間に確認。
  - リリーススコープ（`releases/YYYY-MM.md` の「やる/やらない」）に対し、実装が進んだ項目は進捗・完了として反映。完了したものは振り返り・持ち越しに整理。
  - ドキュメント化されていない重要な実装（新機能・新コンポーネント・外部依存・API変更）→ 対応する `entities/` か `concepts/` を雛形から作成 or 更新。
- **コード↔Wikiの相互参照を張る**: 関連 Wikiページ本文に、対応する実装パスをコードスパンで明記する（例: 「実装: \`src/auth/service.ts\`」）。逆に、必要なら長命なコードコメント等から Wiki ページを参照してよい（プロジェクト規約に従う）。**ディレクトリへの WikiLink は張らない**（パスはコードスパンで書く）。
- 出所のない断定はしない。コードから読み取れないこと（意図・背景）は「未確認」と明示し、人間に確認する。

### 5. フロントマターを更新する
- 触れた各 Wikiページの `updated` を当日の実日付にする。
- 出所が伴う追記なら `sources` を補う。コード変更を根拠にした更新は、本文に対象コードパスを明記してトレーサビリティを保つ。

### 6. index / overview を同期する
- `wiki/index.md` の記載と `wiki/` の実ファイル（`_` 接頭辞を除く）が一致するよう、追加・改名・削除を反映する。全体像やthesis、現在のリリースサイクルに影響するなら `wiki/overview.md` を更新。

### 7. 検索インデックスを更新する（qmd 利用時）
- qmd を導入済みなら `qmd update`（再インデックス）→ `qmd embed` の順で実行する（`qmd embed` だけでは新規・改名ファイルが検索に入らない）。未導入なら `/qmd-setup` を案内。

### 8. log に追記する — `wiki/log.md` の末尾に:
```
## [YYYY-MM-DD] sync | 手編集・実装変更の取り込み
- Wiki編集: [[...]]（M/A/D/R）
- コード変更: `path/...`（→ 反映した [[...]]）
- 波及: リンクN件 / index同期 / 矛盾(ドリフト)N件
- 次の問い: ...
```
（実装上の重要な意思決定を残した場合は別途 `<op>` = `decision` のエントリも追加してよい）

### 9. 報告する
触れたページ・コードを列挙し、未解決のドリフト・要確認のリンク/矛盾・次に調べるべき問いを提案する。

## 便利コマンド
- 変更ファイル一覧: `git status --porcelain`（コミット済みなら `git diff --stat <ref>`）
- 変更本文: `git diff`（または `git diff <ref>`）
- コード変更に対応するWiki探索: `grep -rn '<モジュール/機能名>' wiki`
- WikiLink 抽出: `grep -rno '\[\[[^]]*\]\]' wiki | sort -u`
- ダングリング/被リンク調査: `grep -rn '\[\[対象名\]\]' wiki`
- 規約違反リンク（パス付き）: `grep -rno '\[\[[a-z]*/' wiki`

## 原則
- 手編集・実装の内容は尊重する。勝手に意味を変えず、波及（リンク・整合・記帳・橋渡し）に徹する。
- コードとWikiが食い違うときは、どちらが現実かを切り分けて提示する。**矛盾は消さずに明示**。内容判断（ADRの是非、ページ統廃合）は人間に確認する。
- 軽微で明確な修正（リンク追加、index同期、フロントマター補完、リンク正規化、コードパスの相互参照追記）はまとめて適用してよい。
- 日付は今日の実際の絶対日付を使う。Wikiの詳細規約は `wiki/_schema.md`、コードの規約は `CLAUDE.md`「コードベースの規約」に従う。
- 広範な健全性点検（孤立ページ・陳腐化の全体洗い出し）は `/wiki-lint`、コードの正しさレビューは `/code-review` に引き継ぐ。
