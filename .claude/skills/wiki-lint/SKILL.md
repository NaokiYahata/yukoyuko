# wiki-lint — 健全性チェック

Wikiが育つにつれ溜まる「歪み」を点検し、健康に保つ。**開始前に `wiki/_schema.md`（規約）と `wiki/index.md`・`wiki/log.md` を読むこと。**

## 点検項目

1. **矛盾** — ページ間で食い違う主張。新しい/信頼できる根拠はどちらか。
2. **陳腐化（stale）** — 新しいソースに更新された古い主張。`status: stale`/`superseded` 付与の候補。
3. **孤立ページ（orphan）** — どこからもリンクされていないページ。index 突き合わせやリンク抽出で検出。
4. **欠落概念** — 何度も言及されるのに専用ページが無い概念・実体。
5. **欠けた相互参照** — 関連し合うのにリンクされていないページ。
6. **index / log のズレ** — `wiki/` の実ファイル（`_` 接頭辞を除く）と `index.md` の記載が一致しているか。
7. **規約違反** — フロントマター欠落（`type`/`updated`/`sources`）、パス付きWikiLink（`[[releases/..]]` のようなパス付き）やディレクトリリンクなど `wiki/_schema.md` 違反。
8. **リリース整合** — `release: YYYY-MM` タグが指すリリースハブ（`releases/YYYY-MM.md`）が存在するか / `status: released` のリリースに未完スコープが残っていないか / backlog 項目の陳腐化（既にリリース済み・不要になった項目）。
9. **データギャップ** — Web検索や追加ソースで埋められそうな空白。

## 進め方

- 検出結果を重要度順に一覧化し、それぞれに **具体的な修正案** を添える。
- 軽微で明確な修正（リンク追加、index 同期、フロントマター補完、リンク形式の正規化）は確認のうえまとめて適用してよい。
- 内容判断を伴う修正（矛盾の解消、ページ統廃合）は人間に確認する。
- **次に調べるべき問い** と **取り込むと良いソース** を提案する。
- 結果を log に記録:
  ```
  ## [YYYY-MM-DD] lint | <範囲>
  - 検出: 矛盾N件 / 孤立N件 / ...
  - 対応: ...
  - 次の問い: ...
  ```

## 便利コマンド
- 直近の活動: `grep "^## \[" wiki/log.md | tail -10`
- 全ページ一覧（メタ除く）: `find wiki -name '*.md' -not -path '*/_*'`
- WikiLink 抽出: `grep -rno '\[\[[^]]*\]\]' wiki | sort -u`
- 規約違反リンクの検出: `grep -rno '\[\[[a-z]*/' wiki`（パス付きリンクが無いか）
