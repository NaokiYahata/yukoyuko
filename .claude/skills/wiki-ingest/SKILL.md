# wiki-ingest — ソース取り込み

`raw/` の新規ソースを読み、Wiki（`wiki/`）に統合する。**単に要約するのではなく、既存の知識ベースに織り込む** のが目的。1ソースで10〜15ページに波及しうる。

**開始前に必ず `wiki/_schema.md`（規約）と `wiki/index.md`（現在のWiki構造）を読むこと。** ページ作成は `wiki/_templates/` の雛形から行う。

## 手順

### 0a. チャットに貼り付けられたテキストを raw/ に保存する（`raw/` にファイルが無いケース）

ユーザーがソース本文をClaudeのチャットに直接貼り付けた場合（議事録、メール、Slackログ、仕様の断片など、`raw/_originals/` にファイルが存在しない）、**まずその内容を `raw/<slug>.md` として保存し、真実の出所にする**。会話の文脈だけで取り込むと出所が残らないため、必ずファイル化してから先へ進む。

- 保存先: `raw/<slug>.md`（slug は内容を表す小文字ケバブケース）。冒頭に出所メモを付ける:
  ```markdown
  > 出所: チャット貼り付け / 取り込み日: YYYY-MM-DD / 提供者・元媒体: <分かれば記入（例: 〇〇定例 議事録, △△からのメール）>
  ```
- 貼り付け内容はできるだけ原文のまま保存する（要約・改変しない。整形は見出し付与など最小限に留める）。元媒体・日付・関係者が分かれば出所メモに併記し、トレーサビリティを保つ。
- 保存したら以降は通常どおり、この `raw/<slug>.md` を対象に手順1へ進む。
- ごく短い断片や使い捨ての問い合わせは取り込み不要なこともある。Wikiに残す価値があるかは手順2で人間と確認する（明示の取り込み指示があればそのまま進める）。

### 0b. バイナリ原本のテキスト抽出（pptx / pdf / xlsx / docx 等）

元データがバイナリの場合、まずテキストを抽出して `raw/` に markdown として保存する（**バイナリ原本はGit管理しない**。`.gitignore` 済み）。

- 抽出対象と使うスキル:
  - `.pdf` → **pdf** スキル
  - `.pptx` / `.ppt` → **pptx** スキル
  - `.xlsx` / `.xls` / `.csv` → **xlsx** スキル
  - `.docx` / `.doc` → **docx** スキル
- 抽出先: `raw/<slug>.md`（slug は小文字ケバブケース）。冒頭に元ファイル名・抽出日のメモを付ける:
  ```markdown
  > 抽出元: `raw/_originals/<元ファイル名>` / 抽出日: YYYY-MM-DD
  ```
- 表やスライド構造はできるだけ保持する（xlsxはシート単位、pptxはスライド単位の見出しなど）。
- 既にテキスト/markdownのソースは抽出不要。そのまま `raw/<slug>.md` を対象にする。

### 1. 対象を特定する
- チャットに本文が貼り付けられている場合は手順0aで保存した `raw/<slug>.md` を対象にする。
- 引数でソースが指定されていればそれを使う。なければ `raw/`（と `raw/_originals/`）を見て、まだ `wiki/sources/` に要約が無いソースを探す。複数あれば、どれから取り込むか確認する。
- 抽出した（または既存の）`raw/<slug>.md` を読む。画像を含む場合は本文を先に読み、必要に応じて `raw/assets/` の画像を別途 Read で確認する。

### 2. 要点を議論する（人間と協調）
- 読み終えたら、主要な発見・このプロジェクトにとっての意味・既存の理解との関係を3〜6点で提示し、強調すべき点を人間に確認する。一括取り込みを明示的に指示されている場合はこの確認を省略してよい。

### 3. ソース要約ページを作る — `wiki/sources/YYYYMMDD-<short-title>.md`
- `wiki/_templates/source.md` を雛形に使う。`raw/<slug>.md`（および元ファイル名）を「生ソース」に明記。

### 4. 波及更新する（ここが本体） — 雛形は `wiki/_templates/` を使う
- 登場する人物・組織・システム・機能 → `entities/`（雛形 `entity.md`）を新規作成 or 更新。
- 概念・論点・テーマ → `concepts/`（雛形 `concept.md`）を新規作成 or 更新。
- 意思決定が含まれる → `decisions/`（雛形 `decision.md`）にADRを作成。
- 新しい要求・アイデア → `wiki/backlog.md` に1行追記。進行中リリースのスコープ・状態に関わる内容 → 該当 `wiki/releases/YYYY-MM.md` ハブを更新し、作成したページに frontmatter `release: YYYY-MM` タグを付与（現在のリリースに変化があれば `wiki/overview.md` の「現在のリリースサイクル」も更新）。
- `[[ファイル名]]` 形式（パス無し）で相互参照を張る。更新した各ページの `updated` と `sources` を更新する。
- **矛盾を見つけたら消さずに明示** し、どちらがより新しい/信頼できるか根拠を添える。

### 5. index と overview を更新する
- `wiki/index.md` に新規ページを各カテゴリへ追加（`- [[ファイル名]] — 一行サマリ`）。
- 全体像やthesisに影響するなら `wiki/overview.md` を更新。

### 6. 検索インデックスを更新する（qmd 利用時）
- qmd を導入済みなら、新規mdを検索対象に取り込むため `qmd update`（再インデックス）→ `qmd embed` の順で実行する（`qmd embed` だけでは新規ファイルが検索に入らない）。未導入なら `/qmd-setup` を案内。

### 7. log に追記する — `wiki/log.md` の末尾に:
```
## [YYYY-MM-DD] ingest | <ソースタイトル>
- 抽出: raw/<slug>.md（元: <元ファイル名>）
- 作成/更新したページ: [[...]], [[...]]
- 主な発見: ...
```

### 8. 報告する
触れたページを列挙し、新たに浮かんだ問いや次に取り込むべきソースを提案する。

## 原則
- `raw/` のファイルは抽出後は書き換えない（抽出は新規md作成）。バイナリ原本は触らない。
- 出所のない断定をしない。ソースに無いことは「未確認」と書く。
- 日付は今日の実際の絶対日付を使う。
- リンク・フロントマター等の詳細規約は `wiki/_schema.md` に従う。
